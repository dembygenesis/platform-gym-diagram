Two databases, two operators,
three ways to back them up.

Your cluster runs Postgres and MariaDB, each with its own controller, plus Velero on top. They barely talk to each other — by design. Here's the whole backup landscape, one diagram at a time.

↓ about 6 minutes. Start here.
the one sentence that explains everything Every database is two apps — an operator (the controller, from Helm) and an instance (the actual DB, your YAML). Postgres and MariaDB each have their own operator; they share only plumbing (S3 creds, namespace), not logic. Backups come in three independent layers.

Part 5 — the three layers

Three independent backup systems

Two are database-aware (row-level). The third, Velero, is not — it snapshots files and k8s objects. They cover different things.

rows live in databases; media files live on a PVC — Velero catches the latter.
why WordPress needs two of themA full WordPress restore = MariaDB dump (the posts/settings rows) + Velero (the uploaded images on the wp-content volume). Neither alone is enough. That's why both run at the same 6h cadence.

Part 8 — the scars

What bites, and the reflex

The reflexes to keep across both databases.

operator ≠ instance
Confusing the controller with the database
cloudnative-pg is the brain; apps/postgres is the DB. Deleting the operator app orphans every DB it runs. Know which app you're touching.
# every 2h (cron says 6h)
Comment drift on the MariaDB backup
The header claims 2h; 0 */6 * * * is 6h. Trust the cron, fix the comment — stale docs cause false confidence about RPO.
restore MariaDB → images still 404
Restoring the DB but not the files
WordPress rows live in MariaDB; uploads live on a PVC. A real restore needs the DB dump AND the Velero file backup.
velero backup pg-live
Using Velero as a Postgres backup
Crash-consistent volume snapshots can restore a corrupt DB. Use CNPG's own backup for Postgres; keep Velero for files.
"backups are configured" ✓
Configured ≠ succeeding
CNPG showed a green backup block while archiving failed 100%. Verify a real recovery point exists, don't trust that config is present.
