Bootstrap — the chicken-and-egg, in order

GitOps can run everything except the things GitOps needs to exist first: the cluster, ArgoCD itself, repo read-access, the secret-zero, and stateful data. That's "the normalities." Two phases. The amber steps are the three hard ordering constraints — everything else is order-independent.

PHASE 1 · Terraform · from laptop
1
terraform apply
UpCloud server (2xCPU-4GB, fi-hel1, Ubuntu 24.04) + cloud-init installs k3s v1.34.6+k3s1 + helm. Creates Cloudflare A + wildcard records → 94.237.113.110, grey-cloud (DNS-only).
2
infra ssh-sync --ip …
Repoints the upcloud_infra SSH alias at the fresh box (every recreate = new IP). Then verify: node Ready, helm present.
DNS must be grey-cloud (DNS-only), never proxied. Orange-cloud makes dig return Cloudflare IPs and silently breaks DNS-01 ACME → cert issuance hangs with no error.

The scars — where this class of system bites

The transferable gold. Not commands — reflexes. Each is a failure mode this exact build surfaced, with the move that fixes it. This is the page to re-read before you build #2 by hand.

Stale Terraform lock after an interrupted apply

Error acquiring the state lock · .tfstate.lock.info lingers

An apply killed mid-flight leaves a lock with no process behind it. Don't blindly nuke it — first confirm no terraform is running and no orphaned server exists, or you'll double-provision.

pgrep terraform → check UpCloud server list → terraform force-unlock / rm lock.info → apply

Cert issuance after a rebuild — silent LE rate-limit

Certificate stuck, no error · ACME quota burned

Letting cert-manager re-issue on every rebuild quietly eats Let's Encrypt's weekly quota. Pre-seed the cert secret before root-app so cert-manager adopts instead of issues.

restore-cert.sh BEFORE root-app · verify: 0 CertificateRequests

DB reload that crash-loops the backend

backend: "no schema im_app" · CrashLoopBackOff

The obvious reload (DROP SCHEMA + full restore) deletes the structure migrate owns → backend's migrate-check finds nothing. Restore data-only into the schema migrate built.

pg_dump --data-only --disable-triggers · gate on /health · TRUNCATE+COPY, never DROP

env.ts fails OPEN to production

const apiUrl = NEXT_PUBLIC_API_URL || "https://im.eufit.io"

If the build-arg ever drops, the gym frontend silently calls real prod. A missing config should fail loud, not fall back to the most dangerous default.

assert the build-arg is present at build · or default to a safe/blank base

"No runner pods" looks broken — it's correct

kubectl get pods -n arc-runners → empty

ARC scales to zero. A runner only exists for the life of a job. Empty when idle is the design, not an outage — the whole reason to migrate off always-on systemd runners.

push a commit → watch a runner pod appear, run, and vanish

Wildcard CORS + cross-origin hub→API

access-control-allow-origin: *

The hub calls the backend from the browser across hostnames — only works because CORS is wide open. Fine for a gym; in prod, scope it to the hub origin or the call dies in preflight.

prod: allow-origin = the hub's exact origin, not *

Laptop DNS lies (this very session)

curl → 000 "could not resolve host"

A negative-cached / blocked local resolver returns nothing even when the platform is perfectly healthy. Don't debug the cluster — bypass DNS to prove the platform, then fix the laptop.

curl --resolve host:443:94.237.113.110 … → flush: dscacheutil -flushcache

Stale local gitops clone deploys the wrong thing

Image Updater commits to origin while your local lags

Scripts run from your local copy but ArgoCD syncs origin/main. A divergent local clone is how a clean rebuild once came up with 0 ARC runners.

ALWAYS: git pull --rebase origin main before any setup/teardown
