Part 1 — the everyday loop

What happens when you push code

Six steps. After setup, all six run by themselves.

1
You push to main
DembyOrg/backend_platform
That's the only manual act. Everything below is automatic.
2
A runner pod wakes up
runs-on: arc-hosted
ARC spins a fresh pod for the job, then deletes it. Idle = zero pods.
3
It builds an image
ghcr.io/dembyorg/…:main-<sha>
Tagged by the git commit, pushed to the registry.
4
Image Updater notices
checks every 2 min
It writes the new tag into the GitOps repo as a commit.
5
ArgoCD rolls it out
git is the source of truth
Sees the commit, updates the Deployment to the new image.
6
It's live 🔒
*.test.eufit.io
Serving behind the gateway with a green lock.
ok — that's the whole loop
the one thing to rememberOnly step 4 polls (every 2 min). There's no event for "a new image exists," so something has to look. Every other hop is instant. That 2-min wait is normal, not stuck.
Part 2 — how it gets built from nothing

The setup, in order

GitOps runs everything — except the few things that must exist before GitOps can. You do those by hand, once, in this order.

phase 1
Terraform makes the box
UpCloud + k3s + DNS
One command → a server with Kubernetes and the DNS records pointing at it.
phase 2 · a
Install ArgoCD
the only manual helm install
ArgoCD can't deploy itself, so you plant it by hand. It deploys everything else.
phase 2 · b
Give it the repo key + the cert
read access + pre-seed TLS
So it can read the private repo, and so the cert is adopted, not re-issued.
phase 2 · c
Apply the root app
app-of-apps → 20 apps
Now ArgoCD brings up the whole platform on its own.
phase 2 · d
Unseal the vault + load data
OpenBao, then 25 528 rows
Secrets flow to everything waiting; data lands once the backend is healthy.
Three rules that bite if you flip them:
· cert before the root app (or it calls Let's Encrypt and burns quota)
· vault after the root app (it must be running to unseal)
· data after the backend is healthy (proves the schema exists first)
ok — it's standing up by itself now
Part 3 — how a request reaches an app

From browser to pod

One front door for everything. It sorts by hostname.

Browser opens a URL
hub-gym.test.eufit.io
Hits the one Gateway
:443 · wildcard TLS ends here
Every app shares this single entry point. The lock is verified here.
Sorted by hostname
hub-gym → frontend · backend-platform → backend
Each app has its own little route rule keyed to its name.
Lands on the pod
frontend:3000 / backend:3002
the surprising part you'll forgetThe frontend never talks to the backend server-side. Your browser calls the backend directly, at its own public URL — baked into the page at build time. That's why there are two separate routes, not one clever proxy.
ok — front door, then sort by name
Part 4 — how secrets travel

Vault to app, untouched by hand

The app never knows a vault exists. It just reads a normal secret.

OpenBao holds the truth
the vault · unsealed with one key
Keys live in your password manager — never in git, never in the cluster.
ESO copies it out
openbao-store
Turns a vault path into an ordinary Kubernetes Secret, kept in sync.
The app reads that Secret
envFrom — like any other
No vault client in the app. Swap the vault later and nothing changes.
the neat twistThe CI system's own GitHub key comes through this exact pipeline. The thing that builds every image gets its credential from the vault it helps serve.
ok — vault → plain secret → app

Part 6 — the scars

What bites, and the reflex

The real lessons. Read these before you build the next one by hand.

"no schema im_app" · CrashLoopBackOff
A DB reload that kills the backend
Don't DROP the schema on restore. Load data only, into the schema the app already built.
cert stuck · no error · quota gone
Re-issuing TLS on every rebuild
Pre-seed the cert before the root app, so it's adopted, not re-requested.
apiUrl = ENV || "https://im.eufit.io"
Config that fails open to production
A missing build-arg silently calls real prod. Fail loud, not to the dangerous default.
kubectl get pods → empty
"No runners" looks broken — it's correct
ARC scales to zero. Empty when idle is the design. Push, and watch one appear.
curl → 000 "could not resolve host"
Your laptop's DNS lies
Platform can be perfectly healthy. Bypass DNS to prove it, then fix the laptop.
clean rebuild → 0 runners
A stale local clone deploys the wrong thing
ArgoCD syncs origin, scripts run local. git pull --rebase before every run.
