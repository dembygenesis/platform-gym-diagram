Postgres lesson · read top → bottom

28 GB of Postgres vanished
on one git push.
Here's exactly why.

A real incident on the platform gym: the disk was filling at ~4.5 GB/day. The cause wasn't logs — it was Postgres refusing to throw away files it's designed to protect. Don't study it — just scroll and nod. Each beat fits in your head before the next.

↓ about 5 minutes. Start here.
the one sentence that explains everything Postgres refuses to delete a WAL file until every job that still needs it is done. We accidentally created a job (archiving) that could never finish — so it never deleted anything. The fix wasn't "delete the files." It was cancel the impossible job, which let Postgres's normal cleanup resume.
Part 1 — what we saw

The disk was filling, and it wasn't logs

The obvious suspect (k8s pod logs) was a red herring — 182 MB, nothing. The real weight was in one place.

symptom

df -h / → 63% used, ~6 days left
Left alone, the whole k3s node goes down when the root disk fills.
trace

backend-pg-1 → 28 G
The CloudNativePG Postgres volume. Everything else was rounding error.
shock
The tables were tiny
pgdata/base → 37 MB
37 MB of real data, 28 GB of… something else. That something is WAL.
culprit
1,773 WAL files, stuck
pgdata/pg_wal → 16 MB each
Piling up since the cluster was rebuilt 6 days earlier. Never cleared.
first instinct — and it's wrongThe reflex is "can I just rm them?" No. These files aren't junk — they're Postgres's safety net, and only Postgres knows which are still needed. Delete the wrong one and the database won't recover on its next restart. The rest of this page is why, so you never reach for rm.
ok — 28 GB is WAL, not data
Part 2 — the mechanism

WAL is crash-insurance, and it's mandatory

WAL = Write-Ahead Log. Before Postgres changes real data, it writes down what it's about to do. That order is the whole trick.

1
A write comes in
UPDATE users SET …
Postgres does NOT touch the data file first.
2
It writes to WAL first
pg_wal/00000001…0001
"Write-ahead" = the log entry lands before the data page. Always.
3
Then it updates the data page
in memory, flushed later
The real table change happens after the WAL record is safe.
4
If the server crashes here…
power loss, OOM kill
On restart, Postgres replays the WAL and rebuilds the exact consistent state.
Say this out loud once: you cannot "disable WAL." It's not backup — it's the thing that makes Postgres crash-safe at all. When you hear yourself say "turn off WAL," you mean something else (keep reading).
ok — log first, data second, replay on crash
Part 3 — the janitor

WAL normally cleans up after itself

If WAL is written forever, why doesn't it grow forever? Because of a background job called the checkpointer. This part was never broken — remember that.

1
Every few minutes: a CHECKPOINT
default ~5 min
Postgres flushes all pending changes into the real data files on disk.
2
Now old WAL is redundant
its changes are safely in the tables
Any WAL older than the checkpoint is no longer needed for crash recovery.
3
So Postgres recycles it
deletes / renames for reuse
This is the normal, constant cleanup. It's why pg_wal usually stays small.
the mental hookThe checkpointer is a janitor that only throws away WAL it's sure nobody needs. Under normal life it keeps pg_wal near a small floor (~a few hundred MB). If pg_wal is 28 GB, the janitor isn't lazy — something is telling it those files are still needed.
ok — the janitor keeps WAL small on its own
Part 4 — the twist that caused it all

Backup adds a second lock on the door

Turn on WAL archiving (archive_mode=on) and you get off-box backups + point-in-time recovery. But it changes the deletion rule. Now a WAL file needs two green lights, not one.

deletion = checkpoint AND archived. break one input, the gate is stuck.
the concept everyone missesArchiving has veto power over cleanup. Before archiving, a WAL file was deletable the moment the checkpoint passed it. After archiving, it also has to be safely copied to S3 first. Two locks. Both must open.
ok — backup = a second lock the janitor must wait on
Part 5 — the stuck lock

The backup had been failing since day one

Condition B (archived to S3) was failing 100% of the time, every 60 seconds, since the cluster was born. So no WAL file ever earned its second green light.

1
Cluster was rebuilt 6 days ago
fresh Postgres, same backup config
WAL filenames restart from #1 on a fresh cluster.
2
It reused the same serverName
s3://cnpg/backend-pg-g1
That S3 folder still held WAL from the cluster's previous life.
3
barman ran its safety check
"Expected empty archive"
"This destination already has foreign WAL history — I refuse to write here."
4
Every archive failed → veto stuck
ContinuousArchiving = False
1,773 files, all with a green A, all waiting forever on a red B.
the reframe that makes it clickNothing malfunctioned in the cleanup path. Postgres was doing its job perfectly — correctly refusing to throw away data that hadn't been backed up yet. The bug was upstream (a name collision made backup impossible). The 28 GB was a feature working on top of a broken input.
ok — B could never turn green, so the pile grew
Part 6 — the one-line fix

Remove the backup, and the janitor drains it

We deleted the spec.backup block from the cluster's GitOps YAML. We did not delete a single WAL file by hand. Watch the chain reaction.

1
git push: backup block removed
apps/postgres/cluster.yaml
One file, one commit. This is the entire manual act.
2
CNPG sets archive_mode = off
operator-derived, not a flag
No backup destination → the operator turns archiving off for you.
3
Condition B disappears
the AND-gate loses an input
Now a WAL file only needs condition A again — checkpoint passed it.
4
Next checkpoint: janitor drains
1,773 → 36 segments
All of them qualified for deletion instantly. Recycled in minutes.
before
28 G
1,773 WAL segments
after
577 M
36 segments · disk 63%→27%
"archive_mode" isn't a knob you set. It's on CloudNativePG's fixed-parameter list — the operator owns it. Present spec.backup → on. Absent → off. That's why the lever was delete a block, not flip a setting.
ok — removed the impossible job, cleanup resumed
Part 7 — where to actually look

Three controllers fired, in three places

The trap: you stare at the ArgoCD UI waiting for "WAL cleaned ✅" — it never comes. Argo only knows about config. The real fix happens two layers deeper.

layer 1
ArgoCD — did my config apply?
watch: the Argo UI
Strips spec.backup, prunes ScheduledBackup. Sync flips green. Knows nothing about WAL.
layer 2
CNPG operator — did Postgres accept it?
watch: kubectl get cluster backend-pg
Sets archive_mode=off, does a controlled pod restart. Lives in cnpg-system, invisible in the app tree.
layer 3
Checkpointer — is WAL shrinking?
watch: du -sh pg_wal · df -h /
Pure Postgres internals, no k8s at all. This is the layer that actually reclaims the disk.
the portable lessonGitOps only ever answers "did my declared config apply." It never answers "did the underlying problem resolve." Those are two different questions, checked with two different tools. Confirm the push in Argo, then drop to kubectl/SSH for the truth.
ok — config in Argo, reality in kubectl
Part 8 — the tradeoff you chose

Two uses of WAL — don't conflate them

"WAL" did two jobs here. We kept one and dropped the other on purpose. Knowing which is which is the whole point.

🛟
WAL for crash recovery — KEPT
Local, always on, mandatory. Server dies → replays WAL → consistent. Untouched.
📦
WAL archiving for backup/PITR — DROPPED
Off-box copy to S3, point-in-time recovery. This is what we turned off. It was broken since day one anyway, so we gave up nothing that worked.
🗓️
The replacement — OWED
Plain pg_dump jobs for logical backups. Until those exist, backend-pg has zero off-box backup. Don't leave this "for later."
The real, permanent fix (if you wanted archiving to keep working) was one line: bump serverName: g1 → g2 so the cluster writes to a fresh, empty S3 folder. We chose to drop archiving instead — a valid call, but a choice, not the only option.
ok — crash-safe still; backups now owed
Part 9 — the scars

What bites, and the reflex

The reflexes to keep. Read these before the next Postgres-on-k8s incident.

pg_wal = 28 G · tables = 37 MB
"The database is huge" — no, the WAL is
Always split pgdata/base (real data) from pgdata/pg_wal (the log). A giant pg_wal means archiving is stuck, not that your data grew.
rm 00000001…0042
Deleting WAL by hand to "free space"
Never. Only the checkpointer knows which are still needed for crash recovery. Fix the blocker and let Postgres delete them itself.
"disable WAL"
Confusing WAL with WAL archiving
WAL (crash recovery) can't be disabled. Archiving (backup) can — by removing the backup destination. Different things, different consequences.
serverName: backend-pg-g1 (reused)
A rebuilt cluster writing into old WAL
Fresh cluster + same S3 serverName = collision = "Expected empty archive." Bump the generation marker (g1→g2) on every rebuild.
firstRecoverabilityPoint: None
Backups that never succeeded once
Archiving being "configured" ≠ working. Verify a real recovery point exists — a green config with no successful backup is a silent zero.
staring at ArgoCD for the fix
Expecting GitOps to report disk health
Argo answers "did config apply," never "did the problem resolve." Confirm the push in Argo, verify the result in kubectl/SSH.
