Postgres lesson · diagram-first · read top → bottom

28 GB of Postgres vanished
on one git push.
Here's exactly why.

A real incident on the platform gym: the disk was filling ~4.5 GB/day. The cause wasn't logs — it was Postgres refusing to throw away files it's designed to protect. One diagram per idea. Just scroll and nod.

↓ about 5 minutes. Start here.
the one sentence that explains everything Postgres refuses to delete a WAL file until every job that still needs it is done. We accidentally created a job (archiving) that could never finish — so it never deleted anything. The fix wasn't "delete the files." It was cancel the impossible job, which let Postgres's normal cleanup resume.

Part 2 — the mechanism

WAL is crash-insurance, and it's mandatory

WAL = Write-Ahead Log. Before Postgres changes real data, it writes down what it's about to do. That order is the whole trick.

flowchart TD
  W["A write arrives
UPDATE users SET…"]:::info --> L["1 · Write to WAL FIRST
pg_wal/000…0001"]:::acc
  L --> D["2 · THEN change the data page"]:::info
  D --> X{"server crashes?"}:::warn
  X -->|"on restart"| R["Replay WAL →
consistent state"]:::good
  classDef info fill:#101a24,stroke:#56c2ff,color:#bfe9ff;
  classDef acc fill:#0e2230,stroke:#56c2ff,color:#e8eef4;
  classDef warn fill:#1c1608,stroke:#f4b740,color:#f0dcae;
  classDef good fill:#101c12,stroke:#4ade80,color:#cfe9d6;
log first, data second → any crash is recoverable.
Say it once: you cannot "disable WAL." It's not backup — it's the thing that makes Postgres crash-safe at all. When you say "turn off WAL," you mean something else (keep reading).
ok — log first, data second, replay on crash
Part 3 — the janitor

WAL normally cleans up after itself

If WAL is written forever, why doesn't it grow forever? A background job — the checkpointer. This part was never broken.

the mental hookThe checkpointer only throws away WAL it's sure nobody needs. So if pg_wal is 28 GB, the janitor isn't lazy — something is telling it those files are still needed.

Part 4 — the twist that caused it all

Backup adds a second lock on the door

Turn on WAL archiving (archive_mode=on) for off-box backup + point-in-time recovery, and the deletion rule changes: a WAL file now needs two green lights.

flowchart TD
  A["Condition A ✔
checkpoint passed it"]:::good --> G{"AND"}:::acc
  B["Condition B ✘
archived to S3?"]:::bad --> G
  G -->|"both green"| DEL["Delete the WAL"]:::info
  G -.->|"B stays red"| HOLD["HELD FOREVER
1,773 files pile up"]:::bad
  classDef good fill:#101c12,stroke:#4ade80,color:#cfe9d6;
  classDef bad fill:#1e1013,stroke:#fb7185,color:#f6c9d1;
  classDef acc fill:#0e2230,stroke:#56c2ff,color:#bfe9ff;
  classDef info fill:#101a24,stroke:#56c2ff,color:#bfe9ff;
deletion = checkpoint AND archived. break one input → the gate never fires.
the concept everyone missesArchiving has veto power over cleanup. Before it, a WAL file was deletable the moment the checkpoint passed. After it, the file also has to reach S3 first. Two locks. Both must open.
ok — backup = a second lock the janitor waits on
Part 5 — the stuck lock

The backup had been failing since day one

Condition B failed 100% of the time, every 60s, since the cluster was born. So no WAL file ever earned its second green light.

the reframe that makes it clickNothing malfunctioned in cleanup. Postgres was doing its job perfectly — refusing to discard data that wasn't backed up yet. The bug was upstream (a name collision). The 28 GB was a feature working on top of a broken input.

Part 6 — the one-line fix

Remove the backup, and the janitor drains it

We deleted the spec.backup block from the GitOps YAML. We did not touch a single WAL file by hand.

flowchart TD
  P["git push:
remove spec.backup"]:::acc --> AM["CNPG sets
archive_mode = off"]:::info
  AM --> GONE["Condition B disappears
(gate loses an input)"]:::good
  GONE --> DRAIN["Checkpointer drains
1,773 → 36 segments"]:::good

"archive_mode" isn't a knob you set. It's on CloudNativePG's fixed-parameter list — the operator owns it. spec.backup present → on. Absent → off. That's why the lever was delete a block, not flip a setting.

Part 8 — the tradeoff you chose

Two uses of WAL — don't conflate them

"WAL" did two jobs. We kept one and dropped the other on purpose. Knowing which is which is the whole point.

Part 9 — the scars

What bites, and the reflex

The reflexes to keep. Read these before the next Postgres-on-k8s incident.

pg_wal = 28 G · tables = 37 MB
"The database is huge" — no, the WAL is
Split pgdata/base (data) from pgdata/pg_wal (log). A giant pg_wal means archiving is stuck, not that your data grew.
rm 00000001…0042
Deleting WAL by hand to "free space"
Never. Only the checkpointer knows which are still needed. Fix the blocker and let Postgres delete them itself.
"disable WAL"
Confusing WAL with WAL archiving
WAL (crash recovery) can't be disabled. Archiving (backup) can — by removing the destination. Different things, different consequences.
serverName: backend-pg-g1 (reused)
A rebuilt cluster writing into old WAL
Fresh cluster + same serverName = "Expected empty archive." Bump the generation marker (g1→g2) on every rebuild.
firstRecoverabilityPoint: None
Backups that never succeeded once
"Configured" ≠ working. Verify a real recovery point exists — green config with no successful backup is a silent zero.
staring at ArgoCD for the fix
Expecting GitOps to report disk health
Argo answers "did config apply," never "did the problem resolve." Confirm the push in Argo, verify the result in kubectl/SSH.
